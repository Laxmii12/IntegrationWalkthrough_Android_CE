Logout Tracking And Identity Cleanup
Correct implementation of user logout analytics with smartech.logoutAndClearIdentity(true or false)
Step 1
Track custom event in identified state
->
Step 2
Clear identity with logoutAndClearIdentity(flag)
Why Order Matters
If identity is cleared before your custom event, the user_logout event cannot be attributed to the identified user
Critical Behavior
`smartech.logoutAndClearIdentity()` internally triggers a system logout event, but that internal event is not visible on the dashboard. For analytics and segmentation, explicitly send `user_logout` before clearing identity.
Wrong Sequence
logoutAndClearIdentity() first
trackEvent("user_logout") later
Correct Sequence
trackEvent("user_logout", payload) first
logoutAndClearIdentity(flag) second
Recommended Logout Flow
Call this sequence when user taps the Logout action in your app
1
User taps Logout
App receives a confirmed logout intent from identified user session.
2
smartech.trackEvent("user_logout", eventPayload)
This captures logout in identified state and keeps it visible for dashboard analytics.
3
smartech.logoutAndClearIdentity(true or false)
Then clear user identity and apply your chosen post-logout tracking behavior.
Code Example
Track logout event first, then clear identity
Implementation Pattern
// Called on explicit user logout function handleLogout(currentUser) { const eventPayload = { user_id: currentUser.id, email: currentUser.email, logout_time: new Date().toISOString(), logout_source: "settings_screen" }; smartech.trackEvent("user_logout", eventPayload); // true: stop tracking to previous identity after logout smartech.logoutAndClearIdentity(true); navigateToLogin(); }
Your custom `user_logout` event is what you will use for dashboard reporting and downstream analytics.
Flag Behavior: true vs false
Choose the argument based on whether you want post-logout activity tracking

logoutAndClearIdentity(true)

Use this when you do not want to keep tracking activity after logout against the previous identity.

logoutAndClearIdentity(false)

Use this when you want to continue tracking post-logout activity in your chosen logout-state behavior.

Keep the same ordering in both cases: `trackEvent("user_logout")` first, `logoutAndClearIdentity(flag)` second.
Why Track user_logout First
Business and analytics reasons for explicit custom event tracking before identity cleanup
1. Event attribution remains tied to identified user profile at logout time.
2. You can build reports for logout trends, churn indicators, and session-end behavior.
3. The internal system logout call is not visible in dashboard analytics, so custom event is required.
Complete Logout Handler
Full example with defensive checks
Logout Utility
function onUserLogout(currentUser, trackPostLogoutActivity) { if (!currentUser || !currentUser.id) { return; } const eventPayload = { user_id: currentUser.id, logout_time: new Date().toISOString() }; smartech.trackEvent("user_logout", eventPayload); smartech.logoutAndClearIdentity(!trackPostLogoutActivity); }
In this example, `trackPostLogoutActivity = false` maps to `logoutAndClearIdentity(true)`.
